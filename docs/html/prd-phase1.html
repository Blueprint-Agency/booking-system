01

Product Overview

Problem Statement

Fitness and wellness studios currently rely on generic booking tools that lack flexibility to enforce studio-specific policies, track instructor compensation, or provide meaningful business analytics. Admin overhead is high due to manual attendance tracking, waitlist management, and payment reconciliation.

Product Goal

Build a multi-tenant SaaS booking and operations platform for class-based businesses. Each tenant gets a dedicated subdomain ({slug}.platform.com) with a client-facing booking portal and a full admin backend.

Phase 1 delivers the complete operational core: booking, payments, attendance, CRM, staff management, and reporting.

03

Multi-Tenancy

  • Each tenant is provisioned with a unique subdomain: {slug}.platform.com
  • All data is fully isolated per tenant — no cross-tenant data leakage
  • Admin, staff, instructor, and client accounts are scoped to a single tenant
  • Tenant-level configuration: policies, pricing, branding (logo, colors), notification settings
4.1

Authentication & Accounts

Client Registration

  • Self-registration via email + password
  • OAuth login (Google) — optional at launch, TBD
  • Email verification required before first booking
  • First booking triggers digital waiver flow (see §4.10)

Staff / Instructor / Admin Accounts

  • Created by Admin only — no self-registration
  • Role assigned at creation: admin, staff, instructor
  • Password reset via email link

Session Management

  • Access token expiry: 15 min | Refresh token: 7 days
  • Persistent login (remember me) option
  • Logout invalidates token server-side
4.2

Client Booking Portal

Client-facing interface accessible at {slug}.platform.com.

Session Discovery

  • Default view: list view sorted by date/time ascending
  • Toggle to calendar view (weekly layout)
  • Filters: session category (multi-select), instructor (multi-select), level/tier (multi-select), date range
  • Session card displays: name, category, level, date/time/duration, instructor, spots remaining (real-time), price or "Included in package"
  • Status badge: Open | Waitlist | Full | Cancelled

Booking Flow

1
Select sessionClient opens session detail page/modal
2
Payment path decisionIf active package with balance → deduct from package (skip payment). Otherwise → Stripe Checkout.
3
ConfirmOn successful payment or package deduction → booking created
4
ConfirmationSuccess screen shown + confirmation email sent to client

Waitlist

  • When session is at capacity, "Join Waitlist" CTA is shown
  • Waitlist is an ordered FIFO queue per session
  • On booking cancellation: system auto-promotes the first waitlisted client — creates booking, deducts package or sends payment link, dispatches notification email
  • Clients can remove themselves from waitlist at any time

Cancellation / Reschedule (Client-Side)

  • Accessible from "My Bookings"
  • System checks cancellation policy window before allowing action
  • Within window: booking cancelled, credit returned (package credit or refund)
  • Outside window: penalty applied per policy config (forfeit session or charge fee)
  • Reschedule = cancel + rebook; treated as a new booking for all policy checks

4.4

Payments & Packages

Payment Methods — Phase 1

Stripe only. Credit / debit card via Stripe Checkout. No other payment methods in Phase 1.

Pricing Products

TypeDescription
Drop-inSingle session purchase at fixed price
PackageBundle of N sessions; fixed price; optional expiry date (configurable by admin)
MembershipRecurring monthly subscription via Stripe Subscriptions; grants N sessions/month or unlimited

Purchase Flow

  • Client selects product → Stripe Checkout session created server-side
  • On payment_intent.succeeded webhook: entitlement granted, invoice generated
  • Failed payments: retry handled by Stripe; client notified by email

Package / Membership Logic

  • Package balance decremented on each confirmed booking
  • Expired packages cannot be used; remaining sessions forfeited unless admin manually extends
  • Memberships auto-renew monthly via Stripe
  • Admin can manually add or deduct sessions from a client's package
  • If package is exhausted mid-booking, system falls back to drop-in price
4.5

Invoice System

  • Invoice auto-generated on every successful payment
  • Contains: invoice number, date, line items, amount, payment method, business name/logo
  • Delivered to client's email immediately after payment
  • Stored on client profile; downloadable as PDF
  • Accessible to Admin from client profile or invoice list view
Note: No tax / GST calculation in Phase 1. Line items show pre-tax amounts only.

Recurrence

  • Admin defines recurring sessions via RRULE (weekly, bi-weekly, custom)
  • Editing a recurring session: prompt — "This session only" or "All future sessions"
  • Cancelling a session: all active bookings on that session are cancelled; clients notified by email

Session Types

  • Regular — standard session with recurring support; package-eligible
  • Workshop / Event — one-off; custom pricing; package eligibility configurable per session
4.7

Booking Management (Admin)

  • View roster for any session: client name, booking status, check-in status, package used
  • Manually add a client to a session — bypasses capacity limit (admin override)
  • Manually remove a client — cancellation policy NOT applied
  • Bulk attendance marking: "Mark all Attended" or mark individually
  • Attendance statuses: Pending → Attended | Late | No-Show
  • Filter sessions by date, instructor, category, status

4.9

QR Code Check-In

Check-In Flow

1
Client presents QREach client has a persistent unique QR (encoded with client_id)
2
Staff scansCamera-based scanner in web app; no dedicated hardware required
3
System validatesChecks: booking exists for an active session, late cutoff not exceeded
4
ResultSuccess → status set to Attended or Late. Failure → error message shown (see failure states below)

Late Window Logic

  • late_cutoff_minutes is configurable per session; falls back to global default (default: 10 min)
  • If current_time > session_start + late_cutoff: check-in blocked → status = No-Show
  • If current_time > session_start but within cutoff window: check-in allowed → status = Late

Failure States

ConditionBehaviour
No booking found for this sessionShow error; do not check in
Late cutoff exceededBlock check-in; set status to No-Show; show error
Already checked inShow warning; no status change
4.10

Digital Waiver

  • Waiver content defined by Admin (HTML / rich text)
  • Trigger: first booking or first login — client must complete before proceeding
  • Client must scroll to bottom and click "I Agree" with typed full name (e-signature)
  • Signature record stores: client_id, timestamp, ip_address
  • Signed waiver stored as PDF; accessible from client profile

4.13

Staff Management

Staff Accounts

  • Role staff: front-desk operations only (check-in, view rosters, manual booking add/remove)
  • Admin creates and deactivates accounts; no self-registration

Leave Management

  • Staff / Instructors submit leave requests: type (annual / sick / other), date range, notes
  • Admin sees pending requests in a queue; approves or rejects with optional comment
  • Approved leave reflected on staff calendar
  • Note: System does not auto-block sessions when leave is approved — manual responsibility

Staff Calendar

  • View showing all staff/instructor schedules and approved leave
  • Date range navigation
4.14

Sales & Commission Tracking

  • Each booking / package purchase records the staff_id of the staff member who processed it (manual bookings only); auto-bookings record null
  • Sales report: groupable by staff member, filterable by date range
  • Commission: configurable rate per staff member (% of sale or flat fee per sale)
  • Report is read-only; CSV export available
